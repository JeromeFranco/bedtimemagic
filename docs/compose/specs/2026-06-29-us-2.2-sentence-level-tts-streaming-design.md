# US-2.2: Sentence-Level TTS Streaming — Design Spec

**Date:** 2026-06-29
**Status:** Draft
**User Story:** US-2.2 from `docs/user-stories.md`

---

## [S1] Problem

The current `generate-story-audio` edge function generates ALL TTS audio for every sentence before streaming any data to the client. This means:

1. Client waits for full story audio generation before playback can start
2. No pre-fetch mechanism exists — audio is only fetched when user taps "Play Story"
3. The SSE streaming is just chunked transfer of a pre-generated file, not true streaming

US-2.2 requires: "As LLM streams story text, sentences are extracted at sentence boundaries. Each complete sentence sent to TTS immediately. TTS audio chunks streamed back to client."

## [S2] Solution Overview

**Two-phase architecture with per-sentence TTS streaming:**

1. **Phase 1 — Story Generation** (unchanged): `generate-story` edge function generates full story text via LLM, returns JSON payload
2. **Phase 2 — Audio Streaming** (modified): `generate-story-audio` edge function takes story text, splits into sentences, generates TTS per-sentence, streams audio as each sentence completes

**Key design decisions:**
- Keep `generate-story` and `generate-story-audio` as separate endpoints (preserves story card UX)
- True per-sentence TTS streaming (not batch-then-chunk)
- Progressive file write on client (append each sentence's audio to cache file immediately)
- Pre-fetch first 2 sentences on story card load (background, before user taps Play)

## [S3] Edge Function: Per-Sentence TTS Streaming

**File:** `supabase/functions/generate-story-audio/index.ts`

**Current flow:**
1. Split text into sentences
2. Generate TTS for ALL sentences (batched, concurrent)
3. Combine all PCM into single WAV
4. Stream combined WAV in 32KB chunks via SSE

**New flow:**
1. Split text into sentences
2. For each sentence (concurrent, max 10):
   - Call TTS API
   - On complete → emit `event: sentence` with audio data immediately
3. After all sentences → emit `event: done`

**SSE event format:**
```
event: sentence
data: {"index":0,"total":15,"audio":"<base64_pcm16_wav>"}

event: sentence
data: {"index":1,"total":15,"audio":"<base64_pcm16_wav>"}

...

event: done
data: {"total_sentences":15,"total_bytes":480000}

event: error
data: {"message":"TTS timeout for sentence 5"}
```

**New parameter:** `max_sentences` (optional)
- `{ story_text: string, max_sentences?: number }`
- If provided, only generate TTS for first N sentences
- Used for pre-fetch (first 2 sentences)

**Concurrency model:**
- Use `Promise.allSettled()` with chunk size of 10
- Emit SSE events as each promise resolves (not after all settle)
- Track completed count for progress reporting

**Each sentence's audio is a standalone PCM16 WAV** with its own header, so client can write it independently without worrying about byte alignment.

## [S4] Client: Progressive File Write

**File:** `src/lib/audio-cache.ts`

**Current:** Buffers base64 chunks in `pendingChunks` Map, writes to disk on `finalizeAudioCache()`.

**New:** Write each sentence's audio to disk immediately.

**New function:**
```typescript
writeSentenceToCache(storyId: string, index: number, audioBase64: string): Promise<void>
```

- Decode base64 to bytes
- If index === 0: create new file with WAV header + sentence audio
- If index > 0: append decoded PCM to existing file (skip header)
- File path: `cacheDirectory/audio_{storyId}.wav`

**Changes to existing functions:**
- `finalizeAudioCache(storyId)` — becomes a no-op or just marks file as complete
- `writeAudioChunk()` — deprecated, replaced by `writeSentenceToCache()`
- `pendingChunks` Map — removed (no longer needed)

## [S5] Client: SSE Parsing Update

**File:** `src/lib/audio-stream.ts`

**Current:** Parses `event: chunk` with combined audio data.

**New:** Parse `event: sentence` with per-sentence audio.

**Changes:**
- `streamStoryAudio(storyId, storyText, maxSentences?)` — add optional `maxSentences` parameter
- On `event: sentence`: call `writeSentenceToCache(storyId, parsed.index, parsed.audio)`
- On `event: done`: call `enforceFifoEviction()`, return cache path
- On `event: sentence-error`: log warning, continue (don't fail entire stream)

**Return:** Cache path as soon as first sentence is written (for immediate playback start).

## [S6] Client: Pre-Fetch Mechanism

**File:** `src/lib/audio-utils.ts`

**New function:**
```typescript
preFetchAudio(storyId: string, storyText: string, maxSentences?: number): Promise<string>
```

- Calls `streamStoryAudio(storyId, storyText, maxSentences ?? 2)`
- Returns cache path when first 2 sentences are written
- Called from story card screen (`/story`) on mount

**Integration point:** `src/app/story.tsx`
- On component mount, call `preFetchAudio(story.id, story.storyText)`
- No UI changes — silent background operation
- When user taps "Play Story", first 2 sentences are already cached

## [S7] Client: Streaming Playback

**File:** `src/contexts/PlayerContext.tsx`

**Current:** Creates player from complete audio file.

**New:** Start playback from cache file as soon as first sentence is written.

**Changes to `playStory(story)`:**
1. Call `getAudioSource(story)` — returns cache path if available
2. If cache has data (first sentence written) → create player, start playback
3. If cache empty → show "Preparing audio..." briefly, wait for first sentence
4. expo-audio player loads the WAV file
5. If playback reaches end of written data → player enters buffering state
6. When more data arrives, player resumes automatically (expo-audio handles this)
7. After `event: done`, file is complete — normal seek/controls work

**Edge case:** User taps Play before any audio cached:
- Show brief loading indicator
- Wait for first sentence, then start playback

## [S8] Error Handling

**TTS failure for single sentence:**
- Retry up to 2 times with exponential backoff (1s, 2s)
- If still fails → emit `event: sentence-error` with `{ index, message }`
- Client logs warning, continues playback (audio gap for that sentence)
- Remaining sentences continue processing

**Edge function timeout:**
- Per-sentence TTS timeout: 30s (`TTS_TIMEOUT_MS`)
- If timeout → emit `event: error`, abort remaining sentences
- Client shows error toast, offers retry

**Client disconnection:**
- If client disconnects mid-stream, server continues (no cleanup needed)
- Partial cache file remains — on next play attempt, cache miss → full re-stream

**Cache corruption:**
- If partial write fails mid-sentence, `discardPendingChunks()` cleans up
- On next play attempt, cache miss → full re-stream

## [S9] Testing Strategy

**Edge function tests (`generate-story-audio/index.test.ts`):**
- `splitSentences()` — already tested, keep as-is
- Streaming behavior — verify SSE events emitted per-sentence (not batched)
- `max_sentences` parameter — verify early termination after N sentences
- Error handling — retry logic, timeout behavior, `sentence-error` events
- Concurrency — verify up to 10 concurrent TTS calls

**Client tests:**
- `audio-stream.test.ts` — parse `event: sentence` events, call `writeSentenceToCache`
- `audio-cache.test.ts` — `writeSentenceToCache()` appends to file correctly, handles index 0 (create) vs index > 0 (append)
- `audio-utils.test.ts` — `preFetchAudio()` calls edge function with `max_sentences`

**Integration test (manual):**
1. Generate story → story card loads
2. Verify first 2 sentences cached in background (check file exists in cacheDir)
3. Tap Play Story → audio starts immediately
4. Verify remaining sentences stream and playback continues smoothly
5. Verify cache file grows during playback
